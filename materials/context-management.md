# Context 管理最佳实践

> **用途**：讲师上午讲解和下午小组实战参考
> **核心**：帮助MBA理解"给Agent看什么"比"让Agent看什么"更重要

## 二、外部 Context：你主动给 Agent 的材料

### 2.1 文件上传 vs 文本粘贴 vs 链接提供

| 方式 | 适用场景 | 优点 | 缺点 | 最佳实践 |
|------|---------|------|------|---------|
| **文件上传** | 财报PDF、合同扫描件、长报告 | 保留原始格式、方便引用页码 | PDF解析可能丢失格式、大文件可能截断 | 先转换为纯文本或Markdown再上传；超过50页的先做摘要 |
| **文本粘贴** | 简短材料、关键段落、数据表格 | 格式可控、Agent处理快 | 手动操作繁琐、长文本易出错 | 先用人工筛选出关键段落；用```代码块包裹表格数据 |
| **链接提供** | 新闻文章、官网页面、公开数据 | 直接引用来源、信息最新 | Agent可能无法访问（需要联网权限）、页面结构复杂时解析不准 | 优先提供可访问的静态页面；对关键页面先手动复制主要内容 |
| **结构化数据** | 财务数据、用户数据、实验数据 | Agent易于计算和对比、输出准确 | 需要预先整理成CSV/JSON/表格 | 用CSV格式最佳； column名要清晰；缺失值标注为N/A |

### 2.2 信息密度管理

**核心原则**：Agent 的上下文窗口有限（通常几万到几十万token），信息不是越多越好。

#### 反模式：信息过载

```
❌ 给 Agent 上传：
- 某公司2020-2024年全部年报（5份，每份200页）
- 10篇行业研究报告
- 50条新闻链接

结果：Agent "迷失"在细节中，输出泛泛而谈，抓不住重点。
```

#### 最佳实践：分层投喂

```
✅ 分层策略：

第一层：摘要（人工预处理）
- 用1页纸总结5份年报的核心变化
- 用表格整理10篇报告的核心结论
- 这页纸给 Agent 作为"导航图"

第二层：详细材料（按需深入）
- 当 Agent 需要核实某个数据时，再提供具体页面的文本
- "请查看2023年年报第45页的营收明细表格"

第三层：原始文件（最后防线）
- 当 Agent 对某项数据有疑问时，回到原始PDF核实
- 人做最终守门
```

### 2.3 Context 质量检查清单

在把材料交给 Agent 前，问自己：

- [ ] **相关性**：这份材料和当前任务直接相关吗？有没有"噪音"？
- [ ] **时效性**：数据是最新的吗？Agent 的训练数据截止到什么时候？
- [ ] **完整性**：有没有遗漏关键信息？（如财报只给了利润表，没给现金流量表）
- [ ] **准确性**：来源可靠吗？有没有明显矛盾的数据？
- [ ] **可读性**：格式是否便于 Agent 解析？（PDF表格→纯文本表格）

## 三、对话 Context：历史对话的累积

### 3.1 上下文窗口的概念

> "Agent 能'记住'你们之前说了什么，但这个记忆是有限的。就像你和下属开了一个很长的会，聊到第20个话题时，他可能已经忘记了第1个话题是什么。"

**关键数字**（以 Claude 为例）：
- Claude 3.5 Sonnet：约 200K token（约等于15万字）
- Claude 3 Opus：约 200K token
- 其他模型类似范围

**这意味着**：
- 如果你的对话总长度超过15万字，早期的信息会被"遗忘"
- 对于复杂任务，20-30轮对话后，Agent 可能开始"失忆"

### 3.2 应对"失忆"的策略

#### 策略1：定期重置核心目标

```
用户：请帮我分析宁德时代的财报。（第1轮）
...
（经过15轮讨论，话题已经漂移到了电池技术路线）

用户：回到我们的核心任务——基于2024年Q3财报，评估投资价值。
请忽略之前关于技术路线的讨论，重新聚焦于财务指标分析。
```

> "这不是 Agent 的bug，而是它的设计特性。在长对话中，定期'重置'核心目标，是保持输出质量的关键。"

#### 策略2：关键信息反复提及

```
用户：请记住，我们的核心假设是"产能过剩是短期现象"。
基于这个假设，请重新评估之前提到的3个风险因素。
```

> "重要信息不要只提一次。在关键节点重复核心假设和约束条件，可以对抗'失忆'。"

#### 策略3：拆分长对话

```
会话1：信息收集（让 Agent 读取材料、提取数据）
→ 保存输出

会话2：分析框架（让 Agent 设计分析框架、确定对比维度）
→ 保存输出

会话3：洞察生成（基于前两个会话的输出，生成最终结论）
→ 保存输出
```

> "对于特别复杂的任务，不要试图在一个会话中完成所有事情。拆分成多个会话，每个会话聚焦一个阶段。"

#### 策略4：使用系统 Context（见下节）

把核心规则写入 `CLAUDE.md`，这样即使对话变长，Agent 也会"记得"自己的身份和约束。

## 四、系统 Context：CLAUDE.md / AGENTS.md

### 4.1 什么是系统 Context

系统 Context 是通过配置文件定义的"持久化记忆"。你把规则写进文件，Agent 在每次对话开始时都会读取这些规则。

**类比**：
> "就像公司的《员工手册》。新入职时发一次，但里面的规则长期有效。你不需要每天重复'上班要打卡'，因为手册里已经写了。"

### 4.2 Claude Code 中的 CLAUDE.md

在 Claude Code 中，把 `CLAUDE.md` 放在项目根目录下，Agent 会自动读取。

#### 示例：投资分析团队版 CLAUDE.md

```markdown
# CLAUDE.md

## 角色定位
你是某顶级私募基金投研团队的资深分析师，拥有10年行业研究经验。
你的专长：财务分析、行业趋势研判、投资价值评估。

## 工作风格
- **数据驱动**：每个结论必须有数据支撑，不接受"感觉""可能"
- **审慎保守**：对乐观信号保持怀疑，对风险信号高度敏感
- **结构化表达**：优先使用表格、bullet points，避免大段叙述
- **可追溯**：每个数据点标注来源，每个推断说明逻辑

## 输出规范
- 财务数据：Markdown表格，包含本期、同比、环比三列
- 核心洞察：bullet points，每条不超过40字
- 风险因素：单独成段，用⚠️符号标注
- 总结：150字以内，结论先行

## 禁忌事项
- 禁止编造未公开的财务数据、客户名单、合作协议
- 禁止给出具体的投资建议（买入/卖出/持有/目标价）
- 禁止使用"肯定""必然""绝对"等确定性表述，改用"大概率""存在风险""需关注"
- 禁止基于训练数据中的过时信息做判断

## 校验机制
- 完成分析后，必须列出3个"我最不确定的数据点"
- 如果发现数据异常（如毛利率>100%），标注[数据异常]
- 如果信息不足无法判断，明确说"信息不足"而不是猜测

## 常用术语解释
- ROE：净资产收益率 = 净利润 / 平均净资产
- 经营性现金流：企业日常经营产生的现金流入减去流出
- 同比：与去年同期相比；环比：与上一季度相比
```

### 4.3 系统 Context 的最佳实践

| 实践 | 说明 | 示例 |
|------|------|------|
| **定义角色** | 让 Agent "知道"自己是谁 | "你是资深分析师" vs "你是实习生"，输出质量差异巨大 |
| **明确风格** | 量化风格指标 | "每条洞察不超过40字"比"简洁"更可执行 |
| **列出禁忌** | 明确告诉 Agent 不能做什么 | "禁止编造数据"比"请准确"更有效 |
| **内置校验** | 让 Agent 自我检查 | "完成后列出3个不确定点" |
| **解释术语** | 确保 Agent 理解你的行业黑话 | 在文件中定义常用缩写和术语 |
| **版本管理** | CLAUDE.md 也需要迭代 | 根据实战效果不断调整，保留历史版本 |

### 4.4 系统 Context 的局限性

- **不是万能药**：CLAUDE.md 只能覆盖"通用规则"，具体任务的细节仍需在 Prompt 中说明
- **需要更新**：当项目阶段变化时（如从"研究阶段"进入"汇报阶段"），需要更新 CLAUDE.md
- **不能替代人工**：CLAUDE.md 减少了重复说明，但最终的输出质量仍取决于人的校验

## 五、工具 Context：MCP 与 Skills

### 5.1 MCP（Model Context Protocol）

MCP 是 Anthropic 推出的开放标准，让 Agent 可以安全地访问外部工具和数据源。

**类比**：
> "MCP 就像你给下属配的团队。没有 MCP，他只能靠自己查资料；有了 MCP，他可以调用数据团队、法务团队、市场团队的能力。"

#### 常用 MCP Server

| MCP Server | 功能 | 适用场景 |
|-----------|------|---------|
| **filesystem** | 访问本地文件系统 | 读取/写入项目文件、遍历目录 |
| **github** | 访问 GitHub API | 读取代码仓库、提交issue、查看PR |
| **postgres** | 访问 PostgreSQL 数据库 | 查询结构化数据、生成报表 |
| **brave-search** | 网络搜索 | 实时搜索最新信息 |
| **puppeteer** | 浏览器自动化 | 抓取网页内容、截图 |

#### 配置 MCP（通过 CC Switch）

1. 打开 CC Switch → 点击 "MCP" 标签
2. 点击 "添加 MCP Server"
3. 选择预设模板（如 filesystem）或自定义配置
4. 填写必要参数（如 filesystem 的允许访问路径）
5. 启用同步：选择要同步到哪些 CLI 工具（Claude Code / Codex 等）

### 5.2 Skills（技能）

Skills 是预定义的工具使用流程。你可以把一套常用的 Agent 操作打包成一个"技能"，以后一键调用。

**类比**：
> "Skills 就像 Excel 的宏。你把一套常用的操作录下来，以后点一下按钮就能执行。"

#### Skills 的获取方式

1. **从 GitHub 安装**：CC Switch 支持直接从 GitHub 仓库安装 Skills
2. **从 ZIP 文件导入**：下载 ZIP 包，通过 CC Switch 导入
3. **自定义编写**：自己编写 Skills 配置文件

#### 工作坊中推荐的 Skills

| Skill 名称 | 功能 | 适用场景 |
|-----------|------|---------|
| **market-research** | 自动搜集公开信息、生成对比表格 | 市场调研、竞品分析 |
| **financial-extract** | 从财报PDF中提取关键指标 | 财报分析、投资研究 |
| **meeting-minutes** | 整理会议纪要、提取 Action Items | 会议管理、项目跟进 |
| **bmc-generator** | 基于材料生成商业模式画布 | 战略分析、商业计划 |

## 六、Context 管理的实战原则

### 原则1：质量 > 数量

```
❌ 给 Agent 100页未筛选的材料
✅ 给 Agent 5页人工整理的关键摘要 + 50页原始材料作为"备用查询库"
```

### 原则2：结构化 > 自由文本

```
❌ "这是某公司的信息：他们在做电池，客户有特斯拉，收入挺高的..."
✅ 
```
公司：宁德时代
主营业务：动力电池、储能电池
核心客户：特斯拉、宝马、奔驰、蔚来
2024年Q3营收：XXX亿元（同比+XX%）
```
```

### 原则3：可追溯 > 可信

```
❌ 要求 Agent "确保准确"
✅ 要求 Agent "标注每条数据的来源，不确定时标注[待核实]"
```

### 原则4：分层 > 一步到位

```
❌ 在一个会话中完成：读材料 → 做分析 → 写报告 → 做PPT
✅ 拆分为4个会话，每个会话保存输出，作为下一个会话的输入
```

### 原则5：系统 + 具体 = 最佳

```
CLAUDE.md（系统级）：定义角色、风格、禁忌、校验机制
  +
Prompt（任务级）：定义本次任务的具体目标、Context、格式
  =
高质量输出
```

## 七、Context 管理检查清单

在启动 Agent 任务前，检查：

- [ ] **外部 Context**：我提供的材料是否相关、准确、结构化？
- [ ] **对话 Context**：这个任务是否需要拆分成多个会话？
- [ ] **系统 Context**：CLAUDE.md 是否已定义角色、风格、禁忌？
- [ ] **工具 Context**：需要哪些 MCP/Skills？是否已配置？
- [ ] **信息密度**：材料是否经过人工筛选？有没有信息过载？
- [ ] **时效性**：数据是否最新？Agent 的训练数据是否有盲区？
- [ ] **可追溯**：是否要求 Agent 标注数据来源？

---

## 八、一句话总结

> **管理 Context 的本质，是管理 Agent 的"视野"和"记忆"。给它看对的东西、以对的格式、在对的时机，它的输出质量就会显著提升。**

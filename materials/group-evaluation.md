# 小组评价表

> **用途**：下午小组汇报时的评分参考
> **评分方式**：每组自评 + 其他组互评 + 教授评分，加权计算

---

## 一、评分维度与权重

| 维度 | 权重 | 满分 | 评价要点 |
|------|------|------|---------|
| **Prompt 设计** | 25% | 25分 | 目标清晰、结构完整、约束到位 |
| **Context 管理** | 25% | 25分 | 信息充分、格式合适、无信息过载 |
| **Harness 约束** | 20% | 20分 | 有明确护栏、Agent未跑偏、有校验机制 |
| **输出质量** | 20% | 20分 | 成果可用、无明显幻觉、格式规范 |
| **迭代过程** | 10% | 10分 | 记录了问题、有调整、有Aha Moment |
| **总分** | **100%** | **100分** | |

---

## 二、评分细则

### 维度1：Prompt 设计（25分）

| 评分等级 | 分数 | 标准 |
|---------|------|------|
| **优秀** | 21-25分 | Prompt 包含完整的CO-STAR+Constraint结构；目标具体可执行；Audience和Response明确；有具体的格式和字数要求 |
| **良好** | 16-20分 | Prompt 结构基本完整；目标较清晰；有格式要求；但缺少Audience或Constraint |
| **合格** | 11-15分 | Prompt 有目标和部分结构；但过于简单或模糊；缺少关键的约束条件 |
| **待改进** | 0-10分 | Prompt 只有一句话或几句话；目标模糊；没有结构；没有约束 |

**检查点**：
- [ ] 是否有明确的 Context/背景？
- [ ] 是否有具体的 Objective/目标？
- [ ] 是否有 Style/Tone 说明？
- [ ] 是否有 Audience 定义？
- [ ] 是否有 Response/格式要求？
- [ ] 是否有 Constraint/约束？

---

### 维度2：Context 管理（25分）

| 评分等级 | 分数 | 标准 |
|---------|------|------|
| **优秀** | 21-25分 | 提供的材料高度相关、结构化、经过筛选；使用了多种Context类型（外部+系统+工具）；信息密度适中 |
| **良好** | 16-20分 | 材料相关且结构化；但可能信息略多或略少；Context管理有意识但不够精细 |
| **合格** | 11-15分 | 提供了基本材料；但格式可能不够便于Agent解析；或存在信息过载/不足 |
| **待改进** | 0-10分 | 几乎没有提供Context；或提供的材料与任务无关；Agent只能基于训练数据 |

**检查点**：
- [ ] 提供的材料是否与任务直接相关？
- [ ] 材料是否经过人工筛选（而非原始大文件直接上传）？
- [ ] 是否使用了结构化的数据格式（表格/CSV优于自由文本）？
- [ ] 是否配置了系统Context（CLAUDE.md）或工具Context（MCP）？
- [ ] 信息密度是否适中（不过多也不过少）？

---

### 维度3：Harness 约束（20分）

| 评分等级 | 分数 | 标准 |
|---------|------|------|
| **优秀** | 17-20分 | 有明确的边界约束（只基于给定材料）；有输出格式约束；有校验机制（自我检查）；Agent输出稳定可控 |
| **良好** | 13-16分 | 有基本的约束（如格式要求）；但缺少边界约束或校验机制；Agent偶尔跑偏 |
| **合格** | 9-12分 | 有一些约束意识；但约束不够具体；Agent有时会编造或越界 |
| **待改进** | 0-8分 | 几乎没有约束；Agent自由发挥；幻觉明显 |

**检查点**：
- [ ] 是否有"只基于给定材料"的边界约束？
- [ ] 是否有具体的输出格式要求？
- [ ] 是否有防幻觉机制（如"不确定时标注"）？
- [ ] 是否有校验机制（如"列出最不确定的点"）？
- [ ] Agent 的输出是否稳定（多次执行结果一致）？

---

### 维度4：输出质量（20分）

| 评分等级 | 分数 | 标准 |
|---------|------|------|
| **优秀** | 17-20分 | 输出可直接使用或仅需微调；无明显幻觉；格式规范；逻辑清晰；数据准确（经核实） |
| **良好** | 13-16分 | 输出基本可用；有小部分需要修改；偶有格式不规范；逻辑基本通顺 |
| **合格** | 9-12分 | 输出有可用部分；但需要较多修改；存在一些幻觉或不准确信息 |
| **待改进** | 0-8分 | 输出基本不可用；幻觉严重；格式混乱；逻辑不通 |

**检查点**：
- [ ] 输出是否可以直接用于工作（或仅需少量修改）？
- [ ] 是否有明显的幻觉（编造不存在的数据/事实）？
- [ ] 格式是否符合要求（表格/bullet points/摘要）？
- [ ] 逻辑是否清晰（结论是否有数据支撑）？
- [ ] 关键数据是否经过人工核实？

---

### 维度5：迭代过程（10分）

| 评分等级 | 分数 | 标准 |
|---------|------|------|
| **优秀** | 9-10分 | 详细记录了Prompt的迭代过程（至少2个版本）；清晰说明了问题和调整原因；有Aha Moment（顿悟时刻） |
| **良好** | 7-8分 | 记录了迭代过程；但不够详细；有调整但缺少反思 |
| **合格** | 5-6分 | 有简单的迭代记录；但只记录了"改了什么"，没有"为什么改" |
| **待改进** | 0-4分 | 几乎没有迭代记录；或一次成功（没有尝试和调整） |

**检查点**：
- [ ] 是否展示了至少2个版本的Prompt？
- [ ] 是否说明了每个版本的问题和调整原因？
- [ ] 是否记录了Agent的"翻车"案例？
- [ ] 是否有"原来如此"的顿悟时刻？
- [ ] 是否有可复制的方法论总结？

## 四、评分注意事项

### 对评分人的要求

1. **先听完全部汇报再评分**：避免前几组打分偏高、后几组打分偏低的首因效应
2. **基于事实评分**：根据展示的成果和过程记录评分，不基于"感觉"
3. **鼓励性批评**：扣分的同时，说明"如果这样做可以更好"
4. **关注过程而非结果**：即使最终成果不完美，迭代过程丰富也应给予肯定

### 对教授的建议

- **评分不是目的**：评分的目的是帮助学员理解"什么是好的Agent协作"，不是排名竞争
- **公开评分标准**：在小组实战开始前就公布评分表，让学员有明确的努力方向
- **现场点评优于事后评分**：每组汇报后立即点评，趁热打铁
- **设置趣味奖项**：如"最佳翻车修复奖""最有创意Prompt奖"，降低竞争的紧张感

## 五、自评与互评指南

### 自评（每组给自己打分）

- 如实记录自己的迭代过程
- 不夸大成果，也不低估努力
- 重点反思："如果重来一次，我会在哪里做得更好？"

### 互评（给其他组打分）

- 基于展示的事实评分
- 至少写一条具体优点和一条改进建议
- 避免"人情分"——不要因为关系好就打高分

---

## 六、评分后的反馈

### 即时反馈（每组汇报后）

教授点评1-2分钟：
- 1个亮点（具体）
- 1个改进点（具体）
- 关联三要素（这个案例体现了Prompt/Context/Harness中的哪个？）

### 总结反馈（全部汇报后）

- 公布评分汇总（如果氛围合适）
- 提炼3-5个共性问题
- 展示"最佳实践"案例（征得同意后）
- 强调：评分不重要，重要的是带走方法论

## 七、评分表示例

### 示例：某组的评分

```
小组：第3组
场景：S2 - IC Memo草稿生成（某AI芯片公司A轮投资）

教授评分：
- Prompt设计：22/25（结构完整，但Audience可以更具体）
- Context管理：20/25（材料丰富，但PDF解析时部分表格丢失）
- Harness约束：18/20（边界约束到位，但缺少校验机制）
- 输出质量：17/20（Memo结构清晰，但财务预测部分过于乐观）
- 迭代过程：8/10（记录了2轮迭代，但缺少"为什么改"的反思）
- 总分：85/100

亮点：
- Prompt中使用了"思维链约束"，要求Agent先列出分析框架再填充，输出逻辑非常清晰
- 边界约束明确："禁止给出估值建议"，Agent严格遵守

改进点：
- 可以增加"反面证据呈现"的约束，让Agent主动列出不支持投资的因素
- 迭代记录可以更详细：记录"Agent犯了什么错"以及"我们怎么发现的"

关联三要素：
- 这个案例很好地体现了Harness的价值——有约束的Agent输出质量显著高于无约束
```

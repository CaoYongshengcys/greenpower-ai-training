## 一、三要素一句话总结

| 要素 | 一句话 | 核心问题 |
|------|--------|---------|
| **Prompt** | 给 Agent 的"任务书" | "你要做什么、给谁看、输出什么格式" |
| **Context** | Agent 的"视野"和"记忆" | "你能看到什么、知道什么、记得什么" |
| **Harness** | 让 Agent "不跑偏"的护栏 | "你不能做什么、必须怎么做、不确定时怎么办" |

**核心公式**：
```
驾驭 Agent = 设计好任务（Prompt）+ 管理好信息（Context）+ 设定好规则（Harness）
```

## 三、常见翻车场景 & 应对话术

### 场景1：Agent 编造数据（幻觉）

**表现**：Agent 自信地给出精确但不存在的数字

**应对话术**：
> "大家看这里，Agent 说'预计Q4营收增长15%'——材料里有这个预测吗？没有。它'幻觉'了。这就是为什么边界约束是必须的：'只基于给定材料，不要补充外部信息'。"

**预防措施**：
- Prompt 中加边界约束
- 要求 Agent 标注数据来源
- 人工核实关键数字

---

### 场景2：Agent 跑偏

**表现**：输出和任务无关，或聊着聊着话题漂移

**应对话术**：
> "Agent 现在跑到技术细节里去了，但我们的任务是财务分析。这通常是因为 Prompt 的目标不够聚焦。试试把 Objective 拆成更具体的子任务，或者在一轮对话太长时，重置核心目标。"

**预防措施**：
- Prompt 中 Objective 要具体
- 长对话中定期重置目标
- 用 Harness 约束话题范围

---

### 场景3：Agent 输出太长/太散

**表现**：信息淹没在大段叙述中，无法快速提取要点

**应对话术**：
> "Agent 给了我一篇2000字的分析，但我只有3分钟看。这就是为什么格式约束如此重要：'用表格呈现''每条洞察不超过40字''先给结论再给论据'。"

**预防措施**：
- 指定输出格式（表格/bullet points）
- 限制字数（"每条不超过40字"）
- 要求"结论先行"

---

### 场景4：Agent 遗漏关键信息

**表现**：没提取用户最关心的指标或忽略了反面证据

**应对话术**：
> "Agent 没提现金流的问题，但这是我们最关注的。这通常是因为 Context 投喂不够精准，或者 Prompt 没明确要求'列出所有风险因素'。"

**预防措施**：
- Prompt 中明确列出要提取的指标
- 要求"列出所有风险因素"
- 要求"列出你可能遗漏的关键因素"

---

### 场景5：Agent 语气不对

**表现**：输出太学术、太口语、或不符合受众预期

**应对话术**：
> "这个分析写得像 Wikipedia，但我们是写给投委会看的。记住 Prompt 里的 Audience 要素：'面向投委会，他们熟悉财务术语但时间有限'。同样的分析，给实习生看和给投委会看，应该是完全不同的。"

**预防措施**：
- Prompt 中定义 Audience
- 定义 Style 和 Tone
- 用 CLAUDE.md 统一风格

---

### 场景6：网络/API 故障

**表现**：CC Switch 切换失败，或阿里百炼 API 无响应

**应对话术**：
> "这正是 CC Switch 的价值——切换到备用 Provider。大家看我的操作：托盘菜单 → 选择备用供应商 → 10秒完成切换。"

**预防措施**：
- 提前配置2-3个 Provider
- 现场演示切换流程
- 准备离线数据集作为保底

---

### 场景7：学员说"Agent 没什么用"

**表现**：学员尝试了简单的 Prompt，输出质量低，感到失望

**应对话术**：
> "第一次失望是正常的。Agent 不是魔法，它需要你的引导。试着调整 Prompt——加上 Audience、加上格式约束、加上边界限制——你会发现第二次好很多。"

**引导方法**：
- 现场帮学员改进 Prompt
- 展示改进前后的输出对比
- 强调"迭代"是常态

---

### 场景8：学员过于兴奋，想完全依赖 Agent

**表现**：学员认为 Agent 可以替代人工判断

**应对话术**：
> "Agent 可以帮你写初稿，但签字的是你。最终的责任，永远在人的肩上。Agent 的价值不是'替代分析师'，而是'让分析师把精力放在判断上，而不是搬运数据上'。"

**引导方法**：
- 展示 Agent 的翻车案例
- 强调人工复核的必要性
- 用"杠杆"而非"替代"的比喻

## 四、三要素检查清单（现场用）

### Prompt 检查（问学员）

- [ ] 目标清晰吗？Agent 知道要完成什么具体任务吗？
- [ ] 背景充分吗？Agent 了解前因后果和约束条件吗？
- [ ] 受众明确吗？Agent 知道输出给谁看、用什么语气吗？
- [ ] 格式指定了吗？Agent 知道输出什么形式吗？
- [ ] 约束到位了吗？Agent 知道不能做什么、不确定时怎么办吗？

### Context 检查（问自己）

- [ ] 提供的材料是否相关、准确、结构化？
- [ ] 是否需要拆分成多个会话？
- [ ] CLAUDE.md 是否已定义角色、风格、禁忌？
- [ ] 信息密度是否适中？

### Harness 检查（问学员）

- [ ] 有明确的边界约束吗？
- [ ] 有具体的格式要求吗？
- [ ] 有防幻觉机制吗？
- [ ] 有校验机制吗？
- [ ] 输出是否稳定可控？

## 五、核心教学金句

| 场景 | 金句 |
|------|------|
| 开场 | "Agent 不是来替代你的判断力的，而是来放大你的执行力的。" |
| Prompt | "Prompt 不是提问，而是一份结构化的任务指令书。" |
| Context | "管理 Context 的本质，是管理 Agent 的'视野'和'记忆'。" |
| Harness | "Harness 是安全带，不是枷锁。好的 Harness 让 Agent 更快更准。" |
| 翻车 | "Agent 不'知道'自己不知道。这就是为什么要加边界约束。" |
| 校验 | "永远假设 Agent 会犯错。边界约束、校验机制、人工复核——三道防线。" |
| 总结 | "Agent 是杠杆，不是替代。质量80%取决于你的 Prompt，15%取决于 Context，5%取决于模型。" |

## 六、CC Switch 操作速查

### 添加阿里百炼 Provider

1. 打开 CC Switch → 左侧选择 "Claude Code"
2. 点击 "添加供应商"
3. 选择 "自定义"
4. 填写：
   - 名称：阿里百炼
   - API Key：从阿里百炼控制台复制的 Key
   - Base URL：`https://dashscope.aliyuncs.com/compatible-mode/v1`（根据实际情况调整）
5. 点击 "保存"

### 切换 Provider

**方式1（主界面）**：
- 选择供应商 → 点击 "启用"

**方式2（系统托盘）**：
- 点击托盘图标 → 选择供应商 → 立即生效

### 配置 MCP

1. 打开 CC Switch → 点击 "MCP" 标签
2. 点击 "添加 MCP Server"
3. 选择预设模板或自定义
4. 填写必要参数
5. 启用同步到 Claude Code
